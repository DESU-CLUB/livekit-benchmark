## LiveKit Token Server

`server.js` is implemented and running on port 3000. A copy is saved to `/logs/artifacts/server.js`.

### Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/token` | Issue a new 30-minute JWT |
| `POST` | `/refresh` | Verify an existing token and issue a fresh 30-minute JWT |
| `GET` | `/health` | Liveness check |

### Usage

**Issue a token**
```
GET /token?identity=alice&room=my-room&name=Alice
```
Response: `{ "token": "<jwt>", "ttl": "30m" }`

**Refresh a token**
```
POST /refresh
Content-Type: application/json

{ "token": "<existing-jwt>", "room": "<optional-override>" }
```
Response: `{ "token": "<new-jwt>", "ttl": "30m" }`

**Health check**
```
GET /health
→ { "status": "ok", "service": "livekit-token-server" }
```

### Implementation details

- **TTL**: `'30m'` is passed directly to `AccessToken` options — verified: `exp − nbf = 1800 s = 30 min`.
- **Refresh**: uses `TokenVerifier.verify()` with a 1-minute clock tolerance (grace window for nearly-expired tokens), extracts `sub` (identity), `video.room`, and `name` from the verified claims, then mints a fresh token.
- **Error handling**: 400 for missing/invalid inputs, 401 for token verification failures, 500 for unexpected errors.
- **Credentials**: read from `LIVEKIT_API_KEY` / `LIVEKIT_API_SECRET` environment variables (defaults provided for local dev).
